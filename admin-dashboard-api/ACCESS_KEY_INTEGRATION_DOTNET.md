# Integration Guide: Decoding ERP Access Keys in .NET

This document provides technical instructions for the .NET engineering team on how to decode, validate, and integrate the Access Keys generated by the Admin Dashboard.

## 1. Overview

The `AccessKey` provided to the client is a **JSON Web Token (JWT)**.
Instead of treating the key as an opaque string that must exclusively be sent to the backend for verification, the .NET application can read and extract valuable setup data embedded directly within the key.

This embedded data includes:
- The Client's business identity.
- The specific `modules` the client has purchased or is allowed to access.
- Expiration constraints.

> [!NOTE]
> Extracting the payload does **not** rely on a secret key. Payload claims can be freely read allowing the application to immediately configure its menus and views. However, to guarantee the token was not tampered with, you will still need to cryptographically validate the signature using the shared `JWT_SECRET`.

---

## 2. Token Payload Structure

Upon successful decoding, the JWT payload JSON structure maps equivalently to the following model:

```json
{
  "jti": "5a41cebe-XXXX-XXXX...",
  "clientId": "a3b2c1d0-XXXX-XXXX...",
  "companyName": "Tech Solutions Inc.",
  "email": "owner@techsolutions.com",
  "contactPerson": "Jane Doe",
  "modules": [
    {
        "id": 1,
        "label": "Unified Sales",
        "type": "module",
        "sortOrder": 1,
        "children": [
            {
                "id": 101,
                "label": "Sales",
                "type": "submodule",
                "sortOrder": 1,
                "children": []
            }
        ]
    }
  ],
  "expirationDate": "2026-12-31T00:00:00.000Z",
  "createdAt": "2026-04-03T09:30:00.000Z",
  "exp": 1798675200,
  "iat": 1712136600
}
```

### Claim Definitions:
- `jti`: Unique tracking identifier for this specific token.
- `clientId`: The UUID of the client registered in the central Admin database.
- `modules`: A complex JSON tree structure containing all the modules and submodules explicitly granted to the client. Useful for dynamic UI routing.
- `expirationDate`: A human-readable ISO string of the subscription end date.
- `exp`: Standard Unix timestamp specifying when the token strictly expires.

## 3. Extracting the Data in .NET (C#)

To decode the JWT in a standard modern .NET application, use the official Microsoft library:
**NuGet Package Required:** `System.IdentityModel.Tokens.Jwt`

### Step 3.1: Create strongly-typed Models

Create a set of C# classes to represent the hierarchical token context securely:

```csharp
using System.Text.Json.Serialization;

public class ModuleNode
{
    [JsonPropertyName("id")]
    public int Id { get; set; }

    [JsonPropertyName("label")]
    public string Label { get; set; }

    [JsonPropertyName("type")]
    public string Type { get; set; }

    [JsonPropertyName("sortOrder")]
    public int SortOrder { get; set; }

    [JsonPropertyName("children")]
    public List<ModuleNode> Children { get; set; } = new List<ModuleNode>();
}

public class ClientAccessContext
{
    [JsonPropertyName("jti")]
    public string TokenId { get; set; }

    [JsonPropertyName("clientId")]
    public string ClientId { get; set; }

    [JsonPropertyName("companyName")]
    public string CompanyName { get; set; }

    [JsonPropertyName("email")]
    public string Email { get; set; }

    [JsonPropertyName("modules")]
    public List<ModuleNode> Modules { get; set; }

    [JsonPropertyName("expirationDate")]
    public DateTime? ExpirationDate { get; set; }
}
```

### Step 3.2: Reading Claims directly

If you just need to instantly read the features allowed to provision UI resources without full validation, you can use `ReadJwtToken`:

```csharp
using System.IdentityModel.Tokens.Jwt;
using System.Text.Json;

public ClientAccessContext DecodeAccessKey(string jwtTokenString)
{
    var handler = new JwtSecurityTokenHandler();
    
    // Check if the string is actually a JWT
    if (!handler.CanReadToken(jwtTokenString))
    {
        throw new ArgumentException("Provided access key is not a valid JWT format.");
    }

    var jwtToken = handler.ReadJwtToken(jwtTokenString);

    // Modules sent from the backend are a nested JSON tree
    var modulesList = new List<ModuleNode>();
    foreach(var claim in jwtToken.Claims.Where(c => c.Type == "modules"))
    {
         var parsedNode = JsonSerializer.Deserialize<ModuleNode>(claim.Value);
         if (parsedNode != null)
         {
             modulesList.Add(parsedNode);
         }
    }

    return new ClientAccessContext
    {
        TokenId = jwtToken.Claims.FirstOrDefault(c => c.Type == "jti")?.Value,
        ClientId = jwtToken.Claims.FirstOrDefault(c => c.Type == "clientId")?.Value,
        CompanyName = jwtToken.Claims.FirstOrDefault(c => c.Type == "companyName")?.Value,
        Email = jwtToken.Claims.FirstOrDefault(c => c.Type == "email")?.Value,
        Modules = modulesList,
        ExpirationDate = DateTime.TryParse(jwtToken.Claims.FirstOrDefault(c => c.Type == "expirationDate")?.Value, out var exp) ? exp : null
    };
}
```

## 4. Cryptographic Validation (Recommended)

> [!IMPORTANT]
> If you are blindly trusting the payload to grant access to high-security modules, you **must validate the token signature** first to ensure a user hasn't simply generated their own fake token offline with all modules enabled.

If the ERP app has the shared `JWTKey` environment variable sourced from the Admin Dashboard, use it to fully validate the integrity:

```csharp
using Microsoft.IdentityModel.Tokens;
using System.Text;
using System.IdentityModel.Tokens.Jwt;

public bool TryValidateAccessKey(string accessKey, string sharedKeySecret)
{
    var tokenHandler = new JwtSecurityTokenHandler();
    var key = Encoding.ASCII.GetBytes(sharedKeySecret);

    try
    {
        tokenHandler.ValidateToken(accessKey, new TokenValidationParameters
        {
            ValidateIssuerSigningKey = true,
            IssuerSigningKey = new SymmetricSecurityKey(key),
            
            // Assuming you aren't strictly enforcing Issuer/Audience boundaries across servers yet
            ValidateIssuer = false,
            ValidateAudience = false,
            
            // Standardizes time-expiration evaluation (checks the 'exp' claim on the JWT)
            ValidateLifetime = true,
            ClockSkew = TimeSpan.Zero
        }, out SecurityToken validatedToken);

        return true;
    }
    catch (SecurityTokenExpiredException) 
    {
        Console.WriteLine("Key has expired.");
        return false;
    }
    catch (Exception)
    {
        Console.WriteLine("Key validation failed or signature tampered.");
        return false;
    }
}
```
